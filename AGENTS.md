# Repo 驗證補充規範

本檔補充「什麼情況下才可以回報完成」。

## 完成前證據（強制）

- 禁止只因單一測試通過、單一路徑 UI 正常，就回報功能完成。
- 每個使用者需求都必須先改寫成可驗證的不變量，再確認所有相關 render path / 狀態分支都成立。
- 回報完成前，必須能列出 `需求 -> 驗證證據`；若有任何需求只能靠推論，視為未完成。

## UI / 條件渲染功能（強制）

只要需求涉及顯示 / 隱藏 / filter / section / empty state / 排序 / 分組，必須建立狀態矩陣並逐格驗證。

至少檢查：

- 預設區塊 vs 自訂區塊
- 有資料 vs 被條件清空
- 一般列表 vs 搜尋 / filter 後列表
- 可見項目存在 vs 全部被 hidden / filtered

若修的是其中一個條件分支，必須主動搜尋同元件內的平行分支是否有相同判斷或過濾邏輯，不能只修眼前那一條。

## 回歸測試（強制）

- 產品需求若可寫成不變量，必須補 integration / UI regression test。
- 測試名稱要直接描述產品規則，而不是描述實作細節。
- 與 list / section / hidden 相關的需求，測試必須覆蓋 section 0 與非 section 0。

## 回報格式（強制）

回報完成時，至少要能說清楚：

- 哪個需求
- 由哪個測試或哪個實際驗證證明
- 哪些風險仍未驗證

缺任何一項，不得使用「已完成」「已修好」「可以安裝」這類結論句。
